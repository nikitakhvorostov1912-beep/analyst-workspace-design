<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /llm-config | Untrusted POST/PATCH bodies — Pydantic strict защищает |
| client → /llm-config/test header `X-LLM-API-Key` | API ключ пользователя, проходит через backend для one-shot completion, НЕ персистится |
| backend → external LLM endpoint (httpx) | Outbound HTTP — таймаут 10s обязателен, иначе DoS |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | I (Information disclosure) | GET /llm-config response | mitigate | LLMConfigResponse без поля api_key; проверка тестом `test_post_creates_returns_response_without_api_key` |
| T-05-02 | T (Tampering) | POST/PATCH body | mitigate | Pydantic strict=True + extra=forbid; тест `test_post_strict_rejects_extra_field` |
| T-05-03 | D (DoS) | POST /llm-config/test → outbound httpx | mitigate | Жёсткий timeout=10s в httpx.AsyncClient в test endpoint |
| T-05-04 | E (Elevation) | API key forward через header | accept | Ключ хранится у пользователя (frontend memory/sessionStorage), backend получает только в headers и не логирует тело хедеров; те же гарантии что в Phase 1/3 для /chat |
| T-05-05 | I | error_message в LLMConfigTestResponse может содержать чувствительные данные провайдера | mitigate | Truncate error_message до 200 символов; не пробрасывать httpx response.text напрямую |
</threat_model>
