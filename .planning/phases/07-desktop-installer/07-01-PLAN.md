<objective>
Создать Electron main process (`desktop/main.js`), который связывает backend.exe (wave 1, Plan 7.2) и Next.js standalone frontend (wave 1, Plan 7.3) в одно desktop приложение:
1. Находит два random свободных порта (net.createServer.listen(0)).
2. Spawn-ит backend.exe на порту N с env DATABASE_URL под app.getPath('userData') и BACKEND_ALLOWED_ORIGINS.
3. Spawn-ит node-режим electron.exe (через ELECTRON_RUN_AS_NODE=1) с server.js на порту M, env NEXT_PUBLIC_BACKEND_URL=http://127.0.0.1:N.
4. Ждёт GET http://127.0.0.1:M/ 200 OK через waitForUrl (timeout 30 сек, polling 200ms).
5. Открывает BrowserWindow 1400×900, backgroundColor #0a0a0f, без menu, contextIsolation=true.
6. При close → SIGTERM детям, через 3 сек fallback SIGKILL.

Purpose: Закрыть DIST-01 + DIST-05. Без main.js нет runtime'а, который связывает оба child-процесса в одно desktop приложение и гарантирует их завершение.

Output: `desktop/main.js` (≤200 строк), `desktop/preload.js` (пустой), обновлённый `desktop/package.json` после `pnpm install`.
</objective>

Electron API ключевые:
- `app.whenReady()`, `app.getPath('userData')` (Windows: %APPDATA%/1С Аналитик/), `app.isPackaged`

КРИТИЧЕСКАЯ ТОНКОСТЬ: `process.execPath` в Electron указывает на electron.exe. Чтобы spawnить node-режим (для server.js) надо в env передать `ELECTRON_RUN_AS_NODE: '1'`, иначе electron.exe попытается открыть второе окно вместо запуска server.js. Это известная особенность Electron, документирована.

       - process.on('SIGTERM', () => { cleanup(); process.exit(0); })

<risks>
- **R1 (HIGH): ELECTRON_RUN_AS_NODE забыто.** Frontend spawn откроет второе окно Electron вместо запуска server.js. Mitigation: ENV var обязательно, lint в verify Task 2 (grep tokens).
- **R2 (MED): Child процессы не убиваются если main crash'нулся.** Mitigation: process.on('SIGINT'/'SIGTERM') обработчики + before-quit hook. Orphan handling в Plan 7.5 verification (Task Manager check).
- **R3 (MED): waitForUrl timeout 30s слишком короткий на медленной VM** (backend.exe extract в %TEMP% может занять ~5 сек, frontend cold start ~3 сек, итого &lt;10 сек обычно, но при первом запуске может зайти за 15). Mitigation: 30s даёт x3 запас.
- **R4 (LOW): app.getPath('userData') использует productName из package.json. Если productName содержит русские символы → Windows path с не-ASCII символами.** Это допустимо для SQLite (aiosqlite поддерживает unicode пути), но может удивлять при отладке. Документировать в SUMMARY: точный путь после первого запуска.
- **R5 (MED): Backend startup race — backend.exe может ещё не быть готов когда frontend стартует и пытается NEXT_PUBLIC_BACKEND_URL.** Mitigation: NEXT_PUBLIC_BACKEND_URL — это переменная для browser (renderer), которая используется в runtime через fetch (по факту запроса от user). К моменту первого user-action backend уже готов. waitForUrl ждёт frontend, не backend — это правильно, backend подождёт первого request пользователя.
</risks>
