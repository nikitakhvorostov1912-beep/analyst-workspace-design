Backend coordination через asyncio.Event:
```python
# safety.py
_pending: dict[str, tuple[asyncio.Event, dict]] = {}

def register_pending_confirmation(tool_call_id: str) -> asyncio.Event:
    ev = asyncio.Event()
    _pending[tool_call_id] = (ev, {})
    return ev

def resolve_pending_confirmation(tool_call_id: str, approved: bool) -> bool:
    item = _pending.get(tool_call_id)
    if item is None:
        return False
    ev, payload = item
    payload["approved"] = approved
    ev.set()
    return True

async def wait_for_confirmation(tool_call_id: str, timeout_s: float = 120.0) -> bool | None:
    item = _pending.get(tool_call_id)
    if item is None:
        return None
    ev, payload = item
    try:
        await asyncio.wait_for(ev.wait(), timeout=timeout_s)
        return bool(payload.get("approved", False))
    except asyncio.TimeoutError:
        return None
    finally:
        _pending.pop(tool_call_id, None)
```

```python
class ConfirmRequest(BaseModel):
    model_config = ConfigDict(extra="forbid", strict=True)
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM tool_call execute_code → backend orchestrator | LLM может прислать опасный BSL-код; защита через scan_for_dangerous + явный confirm пользователя |
| Frontend POST /chat/confirm → backend chat router | Любой клиент может POST'ить /chat/confirm; защита через UUID tool_call_id, 404 если не зарегистрирован |
| Browser → backend (CORS) | Origin validation через configurable BACKEND_ALLOWED_ORIGINS; default fail-secure |
| Browser HTML render → injected content | CSP блокирует inline scripts (кроме Next.js hydration whitelist) и third-party connect-src |

<risks>
1. **asyncio.Event between SSE generator and POST endpoint** — оба должны видеть один и тот же _pending dict. Mitigation: модуль-level dict в safety.py; FastAPI один процесс/один loop в dev — работает; для multi-worker uvicorn (prod) — Plan 3.3 в README предупредит «production требует single-worker или Redis-based pending». Документируем явно.
2. **Pydantic strict=True ломает существующие тесты** — у Phase 2 есть тесты POST /sessions/{...}/messages где body нет (тип всё ок). Но есть риск что strict отвергнет string-вместо-числа в `int`-fields. Mitigation: запустить полный backend pytest после Task 1; если что-то падает — фиксить в этом же task (это и есть smoke).
3. **CSP unsafe-inline для script** — Next.js 15 inline hydration требует unsafe-inline для script. Без него белый экран. Mitigation: явно `unsafe-inline` остаётся, документируем как accepted risk (T-03-12).
4. **Dialog @radix-ui/react-dialog нет в зависимостях** — добавляем через pnpm install. Если sandbox не имеет internet — Task 2 не пройдёт. Mitigation: уже есть @radix-ui/react-dropdown-menu (Plan 2.4) → npm registry знаком, должно работать.
5. **DANGEROUS_KEYWORDS false positives** — regex \bЗаписать\( может ловить безобидное `Запись.Записать(...)` в read-only коде. Mitigation: для MVP это ОК — пользователь жмёт «Выполнить» и продолжает; явно документировано в USER.md (Plan 3.4) что dialog появляется на любой совпадающий keyword.
6. **safety.py global state in tests** — пары тестов могут пересекаться через _pending dict. Mitigation: fixture autouse: yield + _pending.clear() в teardown.
</risks>
