| event | data shape |

TableCardPayload:

  meta: {query?: str, duration_ms?: int}

ObjectCardPayload:
  header: {name: str, type: str, path: str}

  forms: list[{name: str, type: str}]
  templates: list[{name: str, type: str}]

  entries: list[{time: str, level: "Info"|"Warning"|"Error"|"Critical", user?: str, event: str, comment?: str}]

ChatRequest расширение (НЕ ломая Phase 1):
  message: str
  session_id: str | None     # если None — orchestrator создаёт новую
  channel_id: str            # REQUIRED в Phase 2 (если нет — 400)

Retry policy:
- network error к LLM/MCP (httpx.RequestError, ConnectError, ReadTimeout) → 1 retry с 200мс delay
- 4xx от LLM/MCP → 0 retry, сразу event:error
- 5xx от LLM/MCP → 1 retry (это transient)
- MCPError JSON-RPC → 0 retry, в tool_result.ok=false + error

Это центр плана. Реализовать `run_chat_loop(db, request: ChatRequest, api_key: str, llm_endpoint: str, llm_model: str) -> AsyncIterator[str]` — async-генератор SSE-строк.

Алгоритм:
1. `ensure_session(db, request.session_id, request.channel_id, title_seed=request.message[:60])` → session_id.
2. `save_user_message(db, session_id, request.message)`.
3. `lookup_mcp_endpoint(db, request.channel_id)` → если None → yield event:error code=unknown_channel; return.
4. yield `format_sse("status", StatusEvent(stage="thinking"))` — это первый байт ≤500мс.
5. Открыть MCPClient(endpoint), `initialize()`, `tools_schema = list_tools()`. Преобразовать MCP tool schemas в OpenAI function format: `[{"type":"function","function":{"name":..., "description":..., "parameters":{...}}}]` (для каждого MCP tool).
6. `messages: list[dict] = [{"role":"system","content":SYSTEM_PROMPT}, {"role":"user","content":request.message}]`. SYSTEM_PROMPT — короткий промпт на русском «Ты аналитик 1С. У тебя есть 10 MCP tools для работы с базой. Отвечай по-русски, краткие TL;DR + структурированные данные через tools».
7. Главный цикл `for iteration in range(MAX_TOOL_ITERATIONS=10):`
   a. yield status thinking. Открыть LLMClient, начать stream_chat_completion(messages, api_key, tools=tools_schema).
   b. Аккумулировать chunks. Каждый chunk имеет `delta = chunk.get("delta", {})`:
      - если в delta есть `content` (str) — yield event:delta {content}; накопить в accumulated_content (для финальной message).
      - если в delta есть `tool_calls: list[{index, id?, function:{name?, arguments?}}]` — накапливать по index в dict accumulated_tool_calls. arguments — это **строка**, частями: склеивать через `+=`.
      - если chunk имеет `finish_reason == "tool_calls"` — финализировать tool_calls: распарсить JSON arguments каждого, добавить в messages как assistant message {role:assistant, content:accumulated_content или null, tool_calls:[...]}; перейти к шагу d.
      - если `finish_reason == "stop"` — выйти из главного цикла.
   c. После закрытия LLM stream — если tool_calls пустые и accumulated_content есть → выходим из цикла (нормальный финал).
   d. Для каждого финализированного tool_call:
      - yield event:tool_call {id, name, args} (args — уже dict после json.loads).
      - yield event:status calling_tool.
      - засечь start_ts, вызвать `mcp.call_tool(name, args)`:
        - retry 1 раз на httpx.RequestError или 5xx;
        - на MCPError или 4xx → result_ok=False, result_error=str(exc).
      - duration_ms = int((time.monotonic()-start_ts)*1000).
      - yield event:tool_result {id, ok, result, error, duration_ms}.
      - если ok и tool name in {"execute_query", "get_object_by_link", "get_metadata", "get_event_log"} — вызвать `build_card_from_tool_result(tool_name, args, result)` → если возвращает payload — yield event:card {type, payload} и накопить card в accumulated_cards.
      - добавить в messages: {role:"tool", tool_call_id:id, content:json.dumps(result, ensure_ascii=False)[:50000]} (cap 50KB на payload, остальное обрезать с маркером ...truncated).
   e. yield status formatting перед след итерацией.
8. После цикла: если iteration == MAX-1 без finish_reason==stop → yield event:error code=tool_loop_limit message="Превышен лимит вызовов tools (10)"; return.
9. `total_duration_ms = int((time.monotonic()-loop_start)*1000)`; `message_id = save_assistant_message(db, session_id, accumulated_content, accumulated_tool_calls, accumulated_cards, total_duration_ms)`; `touch_session`.
10. yield event:done {message_id, total_duration_ms}.

Все exceptions (вне retry-able) → yield event:error и graceful return (не пробрасывать наверх, иначе StreamingResponse развалится).

<risks>
1. **tool_calls в streaming chunks теряются между chunks** — OpenAI-формат возвращает arguments частями по символам в `delta.tool_calls[i].function.arguments`. Если забыть accumulate по index — получим невалидный JSON. **Митигация:** `test_loop_tool_args_accumulate_across_chunks` явно проверяет accumulation.
2. **MAX_TOOL_ITERATIONS infinite loop** — если LLM «зациклится» (каждый ход — новый tool_call с тем же tool name) — pipe никогда не закроется. **Митигация:** hard cap 10, event:error code=tool_loop_limit + явный тест.
3. **MCP result size взорвёт LLM context** — execute_query может вернуть 10К строк. **Митигация:** в loop при добавлении в messages обрезаем content до 50000 символов с маркером "...truncated", оригинал в card.payload идёт целиком в frontend.
4. **SSE генератор бросает exception → StreamingResponse рвётся посередине** — FastAPI плохо обрабатывает exceptions внутри async generator. **Митигация:** все потенциально опасные участки внутри try/except → yield event:error → return (а не raise).
5. **channel_id required ломает существующие фронт-тесты** — Plan 1.2 фронт не передаёт channel_id. **Митигация:** в Plan 2.4 фронт обновится; в этом плане frontend types.ts только сигнатуру меняем, а fetchChat обновляется в Plan 2.4. На переходный период оставить fetchChat как есть (он сейчас не передаёт channel_id, но E2E test не делает реальных запросов).
6. **MCP tool schema не совпадает с OpenAI function format** — у MCP схема в JSON Schema формате, OpenAI ждёт {type:"function",function:{name,description,parameters}}. **Митигация:** explicit conversion в loop step 5, простая обёртка вокруг tools.
</risks>
