# Phase 3 — Information Architecture

**Дата:** 2026-05-13
**Метод:** маппинг 6 user-stories × 10 MCP tools → work-modes → объекты системы → переходы

---

## Принципы IA

1. **Work-mode ≠ экран.** Аналитик работает в **режиме** (Discovery / Bug-triage / Drill-down...), не на «странице». Один экран может обслуживать несколько режимов, переход между режимами — кнопка / шорткат
2. **Проект — главная организационная единица.** Всё (запросы, заметки, объекты-фавориты, requirements) живёт внутри проекта. Проект = одна база 1С на одном `?channel=`
3. **Объект 1С — first-class citizen.** Документ / справочник / регистр имеет «карточку объекта» которая агрегирует данные с 4-5 MCP tools. Все остальные части UI ссылаются на объекты, а не на абстракции
4. **Журнал регистрации = таймлайн.** Всегда есть рядом, всегда фильтруется по текущему контексту. Это «что происходило»
5. **Запрос = переиспользуемый артефакт.** Любой `execute_query` можно сохранить, назвать, повторить, привязать к проекту/объекту
6. **AI-чат — поверхностный слой.** Дёргает MCP, возвращает таблицы/карточки, не «выдумывает» ответы. Каждое утверждение AI — со ссылкой на источник (запрос / журнал / метаданные)

## 7 Work-modes

```
1. DISCOVERY    — «что в этой базе и как устроено»
2. INVESTIGATE  — «вот цифра, разобрать её до строк-источников»
3. AUDIT        — «найти проблемы в коде/данных/правах»
4. TRIAGE       — «вот баг, дойти до причины»
5. MAPPING      — «загрузить внешние данные в 1С»
6. COLLABORATE  — «вести требования с заказчиком»
7. KNOWLEDGE    — «копить и доставать находки»
```

### Mode 1 — Discovery

**Цель:** понять незнакомую базу / новый объект.
**Стартует:** «Открыть проект», переключение на новый проект, ввод «расскажи про X».

**MCP tools:**
- `get_metadata` (5 режимов) — главный
- `execute_query` агрегаты (heatmap активности)
- `get_event_log` за период (что движется)
- `find_references_to_object` (где используется)

**Действия:**
- Карта типов (summary)
- Список объектов с поиском (list + name_mask)
- Карточка объекта (detail + sections)
- Heatmap активности за период
- Расширения и что в них

**Слой 2 наддача:**
- Кеш metadata (SQLite, TTL)
- «Ловушки конфигурации» auto-check для известных конф (УСО/УТ/КА/ERP)

**Слой 3:**
- Bookmark/note на объекте («это ОПП — главный документ поручительств»)
- «Карта моего понимания проекта» — граф ключевых объектов с заметками

### Mode 2 — Investigate (Drill-down)

**Цель:** от цифры → к строкам → к причине → к ответу.
**Стартует:** «Создать ячейку запроса», NL-input, click on cell.

**MCP tools:**
- `execute_query` — главный (с params + include_schema)
- `get_metadata` — для подсказчика
- `get_object_by_link` — drill-down к конкретной записи
- `find_references_to_object` — «эта цифра связана с чем ещё»

**Действия:**
- NL-input → подсказчик → запрос → таблица
- Click на ячейку → новая ячейка с подставленным фильтром (drill-down)
- Save запрос как «именованный» в проекте
- AI explain — TL;DR для руководителя

**Слой 2:**
- Notebook UI (ячейки запросов + результатов + графиков)
- NL→query генератор (читает metadata + syntax_help)
- Auto-detect драйверов цифры (что больше всего влияет)

**Слой 3:**
- Snapshot цифры (permalink + дата + комментарии)
- Compare со снепшотами (сегодня / неделя / месяц)

### Mode 3 — Audit

**Цель:** найти проблемы (код / данные / права / производительность).
**Стартует:** «Запустить аудит проекта», runtime checks.

**MCP tools:**
- `get_metadata` (структура)
- `execute_query` (аномалии в данных)
- `get_access_rights` (права + RLS gaps)
- `get_event_log` (ошибки / медленные)
- `get_bsl_syntax_help` (стандарты)

**Действия:**
- Чек-листы аудита (методология ТКВ / ГОСТ / 1С:ПМ)
- Аномалии в данных (дубли, пустые обязательные, неконсистентность)
- Права: «кто может удалить контрагентов»
- Производительность: «самые медленные операции» из журнала

**Слой 2:**
- Aggregator чек-листов
- Сравнение с эталоном (типовая vs клиентская)

**Слой 3:**
- Аудит-report (PDF/markdown/docx)

### Mode 4 — Triage (Bug-triage)

**Цель:** симптом → причина → факт → задача разработчику.
**Стартует:** «Зарегистрирован баг», вход в инцидент.

**MCP tools:**
- `get_event_log` (главный) — что упало, когда, у кого
- `get_object_by_link` — проблемный объект целиком
- `execute_query` — воспроизведение / поиск похожих
- `find_references_to_object` — что трогает этот объект
- `execute_code` — repro в дев

**Действия:**
- Создать «инцидент» (привязать к описанию бага)
- Найти записи журнала по фильтру
- Открыть проблемный объект
- Найти похожие случаи
- Build bug-card → handoff разработчику

**Слой 2:**
- Bug-card aggregator (журнал + объект + ссылки + repro)
- Confidence scoring (как в QA-аудите 2026-05-05)

**Слой 3:**
- История инцидентов проекта
- Шаблоны handoff-задач для разработчиков

**[Open question]:** где BSL-код модулей? — решится в Phase 4.

### Mode 5 — Mapping

**Цель:** внешний файл (CSV/Excel/API) → документы 1С с валидацией.
**Стартует:** drop файла, «Новая загрузка».

**MCP tools:**
- `get_metadata` (структура целевого)
- `execute_query` (dedup-проверка, поиск контрагентов)
- `execute_code` (создание объектов)

**Действия:**
- Загрузить файл → автодетект колонок
- Mapping wizard (колонка → реквизит, с подсказками типов)
- Type-check (предупреждение о type-mismatch — как с ОПП!)
- Dry-run: «будет создано X, пропущено Y, ошибок Z»
- Apply + лог

**Слой 2:**
- Парсер CSV/Excel/JSON
- Mapping engine + dry-run
- Reusable mapping profiles

**Слой 3:**
- История загрузок проекта
- Шаблоны mapping'ов (APIСигма → ОПП, ЭДО → Реализация, ...)

### Mode 6 — Collaborate

**Цель:** одна точка правды для requirements / вопросов / комментариев заказчика.
**Стартует:** «Открыть требования», переход к встрече.

**MCP tools:**
- `get_metadata` (для ссылок «как сейчас в типовой» / «как в этой базе»)

**Действия:**
- Tree требований (FR/NFR/IR) с MoSCoW
- Open-questions panel
- Comments на каждом
- Templates (методология обследования)
- Auto-agenda для встречи

**Слой 2:**
- Requirements parser (из .docx обследования)
- Diff с эталонной методологией

**Слой 3 (полностью):**
- Persistence + версионирование
- Sharing (link для заказчика, опционально)

### Mode 7 — Knowledge

**Цель:** накапливать находки, ловушки, паттерны со связями.
**Стартует:** capture-button на любом экране, поиск в KB.

**MCP tools:**
- `get_bsl_syntax_help` (для capture'а из BSL)

**Действия:**
- Save заметку (с привязкой к проекту / объекту / запросу)
- Search by tag / fulltext
- «Где я уже сталкивался» — поиск по всем проектам
- Periodic resurfacing

**Слой 3 (полностью):**
- Local SQLite + опционально Obsidian sync (уже есть у P1!)
- Tag system: конфигурация / тема / проект

## Объекты системы (data model)

```
Project (база 1С, channel)
├─ MCP connection (URL, channel, anonymization)
├─ Metadata cache (SQLite, TTL)
├─ Settings (PREFIX, NEW_OBJECTS_IN, ...)
├─ Bookmarks ──────┐
├─ Notes ─────────┐│
├─ Queries ──────┐││ (named, reusable)
├─ Notebooks ───┐│││ (drill-down ячейки)
├─ Incidents ──┐││││ (bug-triage)
├─ Mappings ──┐│││││ (load profiles)
├─ Requirements ──┐│││││
├─ Audit reports ─┐│││││
└─ Sessions (history of actions)
                  └─ Action log: что делали в проекте
```

**Cross-project (global):**
```
Knowledge graph
├─ Configuration (УТ/КА/ERP/УСО)
├─ Object pattern (паттерн ОПП в РТ)
├─ Pitfall (P-001..P-018)
├─ Cross-references (этот pitfall встречался в проектах X,Y)
└─ Tags
```

## UI shell (общая для всех work-modes)

Что есть **всегда** на экране:

| Зона | Содержит | Зачем |
|------|----------|-------|
| **Top bar** | проект (switcher), channel, статус MCP (зелёный/красный), anonymization toggle | mode-агностично |
| **Left rail** | work-mode switcher (7 иконок) + проектные коллекции (запросы, заметки, incidents) | переключение режимов |
| **Main area** | контент текущего mode'а | главное |
| **Right rail** (опц.) | контекстная панель — для текущего объекта (если открыт) или AI-чат | контекст |
| **Bottom strip** | таймлайн журнала регистрации (live + filtered по контексту) | always-visible context |
| **Cmd palette** (Cmd-K) | глобальный поиск + actions | keyboard-driven для P1 |

**Тёмная тема by default.** Контраст высокий. Шрифт — моноширинный для запросов / результатов / журналов; sans-serif для UI текста.

## Переходы между work-modes

```
                   ┌─────────────────┐
              ┌────┤  DISCOVERY      │
              │    └──────┬──────────┘
              │           │
              │           ▼
              │    ┌─────────────────┐
              │    │  INVESTIGATE    │◄────┐
              │    │  (drill-down)   │     │
              │    └──────┬──────────┘     │
              │           │                │
              │           ▼                │
              │    ┌─────────────────┐     │
              │    │  AUDIT/TRIAGE   │     │
              │    └──────┬──────────┘     │
              │           │                │
              ▼           ▼                │
      ┌─────────────────────┐              │
      │  KNOWLEDGE capture  │──────────────┘
      └─────────────────────┘
      
      ┌─────────────┐
      │  MAPPING    │ ◄─── (отдельный flow для загрузок)
      └─────────────┘
      
      ┌─────────────┐
      │ COLLABORATE │ ◄─── (своя «вкладка» для встречи)
      └─────────────┘
```

**Главный путь:** Discovery → Investigate → (если что-то нашёл странное) → Triage → Knowledge capture.
**Боковые потоки:** Mapping и Collaborate — выходят из главного по необходимости.

## Иерархия информации (что выше, что ниже)

```
1. Project switcher                        — самый высокий уровень
2. Work-mode                                — режим работы
3. Main object (если открыт)                — текущий объект 1С
4. Сurrent action (query / read / edit)     — что делаешь
5. Result / state                           — нижний слой
```

**Принцип breadcrumb-навигации:** всегда видно «где я» через breadcrumb сверху:
```
[Проект: Русский Транзит] > [Mode: Triage] > [Документ.ОперацияПоПоручительству] > [Журнал за 30.04.2026]
```

## Что НЕ делаем в IA

- ❌ **Один большой dashboard** — это для менеджера, не аналитика. Аналитик работает в режиме, не на «обзоре»
- ❌ **Папки / категории объектов 1С** — это в самой 1С. У нас — поиск + tags + bookmarks
- ❌ **Notification system** — это в Telegram остаётся
- ❌ **Multi-user / комментарии в реальном времени** — это standalone tool для одного аналитика. Sharing requirements — экспорт link/файла, не collab realtime
