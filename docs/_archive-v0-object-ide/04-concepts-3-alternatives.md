- **Темная тема** by default

# Концепт A. Notebook-first

## Суть

UI как Hex / Observable / Jupyter. **Главный артефакт — ноутбук** (документ с ячейками). Ячейка = запрос / результат / график / AI-объяснение / заметка. Ноутбук персистентен, версионируется, привязан к проекту.

**Метафора:** аналитик «пишет рассказ» из последовательности шагов: вопрос → запрос → ответ → следующий вопрос. UI помогает шаг за шагом.

## Покрытие work-modes

| Mode | Покрытие | Как |
|------|----------|-----|
| Investigate (Drill-down) | **★★★★★** | Notebook — это и есть drill-down: ячейка → click → новая ячейка |
| Audit | **★★★★** | Чек-листы как ноутбуки-шаблоны |
| Discovery | **★★★** | Можно сделать «Discovery notebook» — но это не родная метафора, объект-карточки лучше |
| Triage | **★★★** | Bug-incident как ноутбук, но handoff менее структурирован |
| Knowledge | **★★★★** | Ноутбук-как-заметка с привязкой к запросам естественно |
| Mapping | **★★** | Mapping — это пайплайн, а не notebook. Натянет неудобно |
| Collaborate | **★★** | Requirements не ложатся в ячейки |

## Pros (для моих user-stories)

- **US-3 Drill-down — родная среда.** Кликнул на ячейку 62% → новая ячейка-запрос. Нативно
- **Полностью переиспользуемо.** Ноутбук «РТ-аудит-апрель-2026» = шаблон для мая. Один раз сделал — пользуешься
- **Понятная история мысли.** Можно вернуться через месяц, прочесть как рассказ
- **AI-ячейки органичны** — promt + результат как ячейка, легко ревьюить и переделать
- **Sharing** — экспорт ноутбука в .md / .html для заказчика
- **Git-friendly** — ячейки сериализуются в JSON, легко версионировать

## Cons

- **Discovery новой базы — медленно.** Нет «карты базы» как первого экрана. Каждый раз начинаешь с пустого ноутбука и пишешь Cell 1 «get_metadata summary». Это работа без отдачи 5-10 минут перед получением результата
- **Triage handoff** — слабо. «Передать разработчику» = экспорт ноутбука .md, но там много шума (промежуточные запросы)
- **Объект 1С — не first-class.** Чтобы посмотреть карточку — отдельная ячейка `get_metadata({filter:"Документ.X"})`. Это лишний шаг
- **Mapping** — натягивает: загрузка CSV в notebook это процедурный код, теряется wizard-стиль
- **Параллельные проекты** — каждый ноутбук в своём проекте, переключение медленнее (нужно «открыть проект → найти ноутбук → открыть»)
- **Для P2 (внедренец) — сложновато.** Он привык к Конфигуратору / EDT (древовидная навигация), notebook — это для data-scientist'а

## Ответ на open question BSL

В Notebook концепте: **ячейка-`source-viewer`** для BSL. Аналитик ищет нужный метод через `find_references` или `get_bsl_syntax_help`, добавляет ячейку:
```
source({
  type: "module",
  ref: "Документ.ОПП.МодульМенеджера",
  channel: "edt-mcp"   // ← BSL канал, отдельный от MCP Toolkit
})
```
Source viewer показывает BSL с подсветкой, ссылки на методы → клик добавляет новую ячейку с этим методом.

**Источник BSL:** EDT MCP (Open Question A) — для текущего юзера приоритет, для P2 — расширение позже на (B) git или (C) `get_metadata` если будет API.

## Особенности для P1 / P2

- **P1:** идеально для US-3 (drill-down), знаком с notebook-метафорой (видел Hex, Jupyter). Любит keyboard, ячейки переключаются ⌘↑↓
- **P2:** трудный onboarding — нет дерева метаданных «как в Конфигураторе». Нужны notebook-шаблоны «Старт проекта с УСО» / «Старт проекта с КА» в Library

# Концепт B. Object-centric IDE

## Суть

UI как современная IDE (VS Code, Cursor) **центрированная на объекте 1С**. Слева — дерево / список объектов (через `get_metadata`). Посередине — «карточка объекта» (агрегат 4-5 MCP tools на один документ / справочник / регистр). Справа — контекстный AI-чат + журнал записей этого объекта.

**Метафора:** аналитик «гуляет по базе» как программист по коду. Каждый объект 1С — это «файл» который можно открыть, изучить, выписать заметки, найти ссылки.

## Покрытие work-modes

| Mode | Покрытие | Как |
|------|----------|-----|
| Discovery | **★★★★★** | Tree + карточка объекта = родная метафора. Click → понимание за секунды |
| Triage | **★★★★★** | Объект-центричность идеальна: открыл объект → вкладка журнал → вкладка где использован |
| Audit | **★★★★** | Аудит = «пройтись по объектам», список аномалий в side panel |
| Investigate (Drill-down) | **★★★** | Drill-down есть, но не такой плавный как в notebook — нужно открывать «query view» |
| Knowledge | **★★★★** | Заметки привязаны к объектам естественно («заметка про ОПП») |
| Mapping | **★★★** | Wizard в отдельной view, можно ссылаться на целевой объект |
| Collaborate | **★★★** | Requirements панель с ссылками на объекты |

## Pros (для моих user-stories)

- **US-1 Discovery — пиковая.** Дерево метаданных + карточка с heatmap, расширения видны сразу. P2 (внедренец) — мгновенно ориентируется (это «как в Конфигураторе, но удобнее»)
- **US-2 Bug-triage — пиковая.** Карточка объекта = sum 4-5 MCP tools на один экран. Click на проблемный реквизит → видишь конфликт типов
- **Объект first-class** — bookmarks, заметки, инциденты привязываются к объекту натурально
- **P2-дружелюбен.** Внедренец увидит знакомую модель «дерево метаданных слева»
- **Журнал всегда в подвале** — фильтруется по текущему объекту
- **AI-чат справа** — контекстный, «по текущему объекту». Меньше галлюцинаций (контекст явный)
- **Параллельные проекты** — переключение проекта = подмена дерева слева, остальное shell

## Cons

- **US-3 Drill-down — менее плавно.** Открыть «view запросов» как отдельную панель → запрос → результат → drill-down. В notebook это родная мета, тут — отдельный режим
- **Плотный экран.** Tree + card + chat = 3 колонки + журнал. На laptop 13" будет тесно (один из right-rail элементов придётся скрывать)
- **Notebook-метафора отсутствует** — нельзя «рассказать историю расследования» одним документом. Notebook эмулируется через инцидент с шагами, но менее естественно
- **Mapping** — нужно открывать отдельный экран (wizard), нарушает «единство IDE»
- **Для P1 первый день** — overwhelming density. Нужны collapse / focus mode

## Ответ на open question BSL

В Object-centric концепте: **встроенный BSL-viewer** как панель «⚙ Код» в карточке объекта. Открывает модуль через выбранный BSL-канал (EDT MCP / git / files). Подсветка синтаксиса + click на метод → переход. Если канал не настроен — кнопка «Открыть в EDT» (deep-link).

**Источник BSL:** настраиваемый, **рекомендация:** EDT MCP для P1 (готово), files-via-git для P2 (когда репо доступно).

## Особенности для P1 / P2

- **P1:** знаком с метафорой (VS Code), keyboard-shortcuts (`Cmd-P` по объекту, `Cmd-B` toggle tree). Density-первое — устраивает
- **P2:** идеальная стартовая среда — дерево даёт ощущение «контроля и обзора», карточки разъясняют через hover. **Лучший выбор для P2**

## Суть

**Метафора:** аналитик «беседует с экспертом» который имеет доступ к базе. Спросил → получил структурированный ответ → задал follow-up.

## Pros (для моих user-stories)

- **NL-first нативно.** Подсказчик запросов — это и есть AI в чате
- **Low onboarding** — любой кто знает Claude знает как работать. Идеально для **P2**
- **Knowledge work — пиковая.** История чатов = персональный поиск «о чём я уже спрашивал»
- **AI-objected first.** Все галлюцинации проверены через MCP — каждый ответ ссылается на запрос/журнал
- **Mobile-friendly** — чат на телефоне работает, всё остальное нет
- **Multi-project лёгко** — каждый чат привязан к проекту, список чатов сгруппирован
- **Анонимизация** уместна по умолчанию ON — AI получает токенизированные данные

## Cons

- **US-1 Discovery — слабо.** «Расскажи про базу» → текстовый ответ, не интерактивная карта. Древовидное исследование (как у P2) почти невозможно
- **US-2 Triage — средне.** Bug-card собирается, но handoff менее структурированный — это чат, не документ
- **Контроль теряется.** AI решает что показать, аналитик не видит «полную картину базы»
- **Долгие чаты — каша.** История на 50 сообщений — сложно нагуглить
- **Density низкая.** Для P1 (любит плотные таблицы) — много вертикального скролла
- **Цена.** AI-токены за каждый запрос. С анонимизацией дешевле, но не нулевой
- **Параллелизм** — одно окно чата, перейти к ОПП-расследованию по РТ из чата Век = переключение проектов

## Ответ на open question BSL

В Chat-first концепте: **«inline code card»** — AI получает source через выбранный BSL-канал, показывает фрагмент с line numbers + объяснением. Click «открыть в EDT» = deep-link.

**Источник BSL:** AI-tool по выбору (EDT MCP / files-via-git) — pluggable, аналитик выбирает один раз в настройках проекта.

## Особенности для P1 / P2

- **P1:** будет душить по плотности, любит видеть «целое». Хорош как **сопутствующая поверхность**, но не главная
- **P2:** **очень комфортно.** Знает Claude, диалог естественный, не страшно «спросить глупость». Идеально для onboarding в незнакомую базу

# Гибридные варианты (комбинации)

Концепты не взаимоисключающие. Реалистичные комбинации:

## H1. Object-IDE + inline Notebook view

Главный — Object-IDE. На карточке объекта есть вкладка «Notebook» где можно вести drill-down ячейками по этому объекту. Notebook'и привязаны к объектам или к incidents.

**Покрытие:** все 7 modes хорошо. **Сложность:** высокая (две метафоры). **Recommend для:** P1 + P2 одновременно.

## H2. Object-IDE + AI-chat в right rail

Главный — Object-IDE. AI-chat — постоянно открыта боковая панель, контекст = текущий объект. Чаты пер-объект сохраняются.

**Покрытие:** Discovery/Triage пиковые, Knowledge через chat-history. **Сложность:** средняя. **Recommend для:** P1 (основной) + P2 (видит и tree и chat).

## H3. Chat-first + Object workspace по deep-link

Главный — Chat. Из любого ответа AI («открой документ X») — выпадает «Object workspace» как modal/sidebar поверх чата. Tree метаданных вызывается по `/tree`.

**Покрытие:** Knowledge/Drill-down хорошо, Discovery компромисс. **Сложность:** средняя. **Recommend для:** P2 с переходом на P1 со временем.

# Рекомендация (без выбора, ориентир)

Для **MVP-1 (закрыть US-1 + US-2 + US-3)** оптимальный вариант — **H2 (Object-IDE + AI right-rail)**:
- Покрывает 3 главные user-stories на ★★★★+
- P1 получает density, P2 получает дерево
- AI-чат не главный, но всегда под рукой
- NL→query подсказчик встроен в input запросов и в чат
- Notebook отсутствует — это компромисс для US-3, но drill-down делается через «query view» которая открывается по `Cmd-Q`

Для **post-MVP (US-3 → US-6)** — добавить:
- Notebook-view внутри incidents (для bug-расследований) — слабая версия концепта A
- Knowledge graph как отдельный режим — поверх Chat history

**Но это не выбор** — пользователь должен сам решить. Концепты A и C тоже валидны если приоритеты другие:
- Если важнее US-3 (drill-down) и shareable artifacts — **A. Notebook**
- Если важнее быстрый MVP и low onboarding для P2 — **C. Chat**
