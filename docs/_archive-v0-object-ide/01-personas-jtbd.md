# Phase 1 — Personas + JTBD

**Дата:** 2026-05-13
**Решения (зафиксированы):**
- Multi-tenant: **опция** (как `?channel=` в MCP) — пользователь сам решает работать с одной базой или 5-10
- Анонимизация: **опция** (toggle в UI, по умолчанию OFF на первом запуске)
- BSL-исходники: **открытый вопрос** — решим в Phase 4 на конкретном концепте
- Кэш metadata: **локально** (SQLite, TTL configurable)
- NL→query подсказчик: **в scope**, ключевой AI-блок

## Персона P1 — Никита Хворостов (главная, primary)

### Профиль
- **Роль:** 1С-аналитик/архитектор на постоянке, 4+ года активной работы
- **Бэкграунд:** читает BSL свободно, не пишет руками — проектирует, ставит задачи, ревьюит AI-сгенерированный код через Claude Code
- **Стек:** Win10, EDT 2025.2.5, тёмная тема, русский, без GPU. Платформа 8.3.27+ ↔ 8.5.x на горизонте
- **Параллельные проекты одновременно:** 4-6 разных конфигураций (УТ 11.5, КА 2.5, ERP УСО, КА аудит, миграции БП)
- **AI-savvy:** правило 3 итераций Матакова, не доверяет AI слепо, но активно использует MCP к живой базе

### Контекст работы
- **Где:** удалённо, дома, на встречах с клиентами (онлайн + офис клиента)
- **Когда:** дневные блоки 4-6ч концентрации + встречи разбросаны по неделе
- **С кем:** клиенты (3-5 параллельно), разработчики 1С (внешние, через Telegram), AI (Claude Code)
- **Связи:** заказчик ←→ аналитик ←→ AI ←→ разработчик. Аналитик — центр.

### Цели верхнего уровня
1. Закрыть проект **дешевле** чем при найме разработчика на каждую задачу (AI делает 70% грязной работы)
2. **Быстро** входить в новый проект (день, не неделя) — конкуренция на рынке у тех кто стартует за 3 дня обследования
3. Не терять контекст между параллельными проектами — что обсудили / что осталось / куда вернуться
4. **Не быть узким горлом** — не превратиться в человека-Excel'я, ставить задачи AI и ревьюить, а не «делать»

### Что мешает сейчас
- Telegram-чат с разработчиком = единственный канал handoff. Контекст в чате растворяется через неделю
- При входе в новую базу — день на «понять что задействовано»
- Цифра из отчёта руководству → SQL-запрос → форматирование в Excel = 1-2 часа на каждый раунд вопросов
- AI галлюцинирует по 1С (см. правило 3 итераций) → не доверяю, перепроверяю через MCP
- Параллельные проекты → постоянное переключение контекста, перечитывание собственных заметок в `.md`

### Психологический портрет (важно для UI)
- **Брутальная честность:** не любит «должно работать», требует «проверено и работает»
- **Действие > Discovery:** не любит UI который заставляет много кликать перед получением результата
- **Тёмная тема:** глаза болят на белом фоне после 6 часов работы
- **Авто-приветствия и onboarding-туры — раздражают.** Сразу к делу
- **Любит keyboard shortcuts** — мышью кликать после 4 лет работы не хочется

### JTBD-формулы для P1

| US | Когда я ... | Я хочу ... | Чтобы ... |
|----|-------------|-----------|-----------|
| US-1 Discovery | открываю незнакомую базу клиента или впервые работаю с объектом | за час получить карту «что задействовано / что доработано / как данные текут» через MCP-чтение метаданных и журнала | НЕ читать BSL вручную и НЕ задавать клиенту вопросы, ответы на которые уже в коде |
| US-2 Bug-triage | слышу «документ X упал» / «расширение работает не так» | за минуты пройти путь симптом → запись журнала → код в строке N → данные через запрос → confidence 85% | выдать разработчику задачу с фактом, а не «попробуй посмотреть» |
| US-3 Drill-down | руководитель / клиент показывает цифру | за минуты декомпозировать её через NL → запрос → таблица → drill-down → AI-объяснение | дать ответ той же встречи, а не «через неделю когда разработчик сделает отчёт» |
| US-4 Маппинг данных | мне приносят CSV / выгрузку API / Excel и просят «загрузи как документы» | за один проход увидеть mapping, конфликты, dry-run результат | НЕ узнавать о 296 дублях после прогона и НЕ получить 32 битых документа (как с ОПП 2026-04-30) |
| US-5 Согласование | веду требования с заказчиком | одну точку правды для вопросов / ответов / гипотез / открытых пунктов | перед встречей открыть «что не закрыто» за 30 секунд, а не искать в Telegram |
| US-6 Knowledge | работаю над конфигурацией / темой | складывать находки в knowledge graph со связями и быстро находить «у меня уже было нечто похожее в Веке» | НЕ дублировать research каждый раз с нуля |

### Anti-patterns (что НЕЛЬЗЯ делать в UI для P1)
- ❌ «Вы новенький? Давайте я покажу вам туториал» — игнорировать сразу
- ❌ Светлая тема как default — отвращение с первого экрана
- ❌ Wizards с 5+ шагами на простую операцию — open-source инструменты так и не запоминают
- ❌ Авто-сохранение без визуального индикатора — не доверяет невидимым процессам
- ❌ «Сделаем за вас» без возможности увидеть и поправить промежуточный шаг
- ❌ Английский UI в инструменте под русскоязычные 1С-проекты

## Персона P2 — Антон Демиденко (вторичная, secondary)

> Решение: persona №2 = **внедренец на новом проекте УСО / КА с нуля**. Обоснование: это самый частый тип юзера в 1С-консалтинге, и именно он первый встретит «незнакомую базу» в максимально остром виде. NB: разработчик-аналитик (вариант (в)) близок к P1 и не даёт нового угла. Ведущий аналитик клиента (вариант (б)) — это потребитель отчётов, не пользователь рабочего места.

### Профиль
- **Роль:** ведущий внедренец 1С-франчайзи (или freelance), 2-3 года опыта
- **Бэкграунд:** прошёл сертификацию по типовому функционалу (УТ/КА/ERP), но при первом контакте с конкретной базой клиента всегда чувствует «нет ландшафта»
- **Стек:** Windows, EDT или Конфигуратор (миксует), русский UI обязательно
- **Параллельные проекты:** 1-2 (один основной + хвост от предыдущих)
- **AI-savvy:** на старте слабая. Слышал про Claude, пробовал ChatGPT 1-2 раза, разочаровался («не знает 1С»)

### Контекст работы
- **Где:** офис франчайзи + командировки к клиенту (1-2 раза в неделю)
- **Когда:** жёсткий KPI по часам по конкретному проекту, давление от руководителя «когда сдашь?»
- **С кем:** клиент (заказчик от бизнеса + IT-директор) ←→ внедренец ←→ команда франчайзи (разработчик, методолог)

### Цели верхнего уровня
1. Закрыть **этап обследования** за нормативные 2 недели (а не за 3-4 как сейчас)
2. Не упустить ловушки конфигурации (P-001..P-018 из УСО — нужны как чек-лист на старте)
3. Сдать ТЗ заказчику без переделок — нужны структурные шаблоны (методология ТКВ, ГОСТ)
4. Не зависеть от методолога «дай шпаргалку» — иметь автономный инструмент

### Что мешает сейчас
- Базу клиента видит **впервые** — Конфигуратор покажет «что есть», но не «что задействовано»
- Знает теорию УТ, но конкретные доработки клиента — terra incognita
- AI-инструменты (ChatGPT) не дают полезного по конкретной базе, потому что не имеют доступа
- При написании ТЗ копирует структуру из прошлых проектов — половина пунктов лишняя, какие-то пропущены

### JTBD для P2 (отличия от P1)
- **US-1 Discovery** для P2 ещё критичнее — это **80% его работы** на старте проекта. Чек-лист «что проверить в любой новой базе» — must have
- **US-5 Согласование** — P2 нужны **структурные шаблоны** (методология ТКВ, разделы ТЗ, чек-лист DDТЗ из 168-страничного отчёта Век)
- **US-3 Drill-down** — реже, чем у P1, но критично для финального отчёта руководству заказчика
- **US-2/US-4** — реже, потому что внедренец чаще проектирует, чем правит баги или мигрирует данные
- **US-6 Knowledge** — больше потребляет (читает чужие материалы), меньше создаёт

### Где P1 и P2 совпадают / расходятся

| Аспект | P1 (Никита) | P2 (Антон) |
|--------|-------------|------------|
| Главная боль | контекст-свитчинг между проектами | первый день в незнакомой базе |
| Доверие к AI | среднее, через MCP-валидацию | низкое, нужен авторитет «1С:Напарник» / методологические шаблоны |
| Темп | быстрый, keyboard-driven | средний, готов к wizards |
| UI complexity | принимает density, читает таблицы | нужны explainers, hover-подсказки |
| Скорость учения | моментально (читает doc один раз) | требуется onboarding по новой фиче |
| Параллелизм проектов | 4-6 | 1-2 |

**Следствие для дизайна:** density-by-default (для P1) + **режим «explain me»** который раскрывает подсказки/туториалы по запросу (для P2). Не делать отдельный UI для каждой персоны — это разные «density modes» одного интерфейса.

## Открытый вопрос (передаётся в Phase 4)

**Откуда брать BSL-исходники модулей расширения для US-2 (Bug-triage)?**

Варианты:
- **A) EDT MCP** (port 8770, 34 tools) — даёт чтение модулей в EDT-формате. Минус: завязка на установленный EDT, специфично для русско-транзитного workflow
- **B) Файлы CFE из git-репо** клиента — самый чистый канал, но требует доступ к репо
- **C) `execute_code`** + чтение через `Метаданные.Расширения[N].ПолучитьМакет(...)` — есть ли такой API?
- **D) Не показывать BSL код вообще** — показывать только результат: «модуль X строка Y» как ссылку, открытие в внешнем редакторе

Это решение влияет на концепт. Разберём на Phase 4.
