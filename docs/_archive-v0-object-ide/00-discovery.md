## Группировка → 6 повторяющихся паттернов

Эпизоды кластеризуются в **6 user-stories** (5 чистых + 1 «knowledge work» как объединение редкого documenting/research).

### US-1. Discovery незнакомой базы / нового объекта

**Сырые эпизоды:** 1, 8, 10, 14, и частично 13.

**История:** *Когда я открываю незнакомую базу клиента (или незнакомый объект — документ ОПП, регистр поручительств), я хочу за час получить картину «что есть, что задействовано, что доработано, как данные текут», чтобы не читать BSL/XML вручную и не задавать клиенту вопросы на которые ответы уже в коде.*

**Реальная боль:**
- Эп.1: захват трафика + декомпиляция DLL заняли день — потому что не было способа спросить «как этот объект работает в Сигме»
- Эп.8: 68 документов клиента читал последовательно — нет TL;DR + связей между требованиями
- Эп.14: при миграции «расхождения УТ↔БП почти никогда в данных, а в политиках» — но политики разбросаны по 30+ местам в конфигурации
- Эп.10: бизнес-модель РТ прояснял через 3-4 итерации переспросов клиента

**Что должен делать инструмент:**
- Карта конфигурации: какие подсистемы реально используются (по движениям регистров, не по метаданным)
- Карта доработок: что в типовой / что в расширении / что захардкожено в проведении
- Объект → как используется: где читается, где пишется, какие подписки, какие отчёты
- Bookmark: «эта группа объектов = подсистема поручительств» — с заметками
- AI: «объясни на пальцах что делает регистр Х и почему он движется при проведении документа Y»

### US-2. Bug-triage по живой базе: связать симптом → код → данные

**Сырые эпизоды:** 4, 7, 17 (плюс косвенно 5).

**История:** *Когда мне говорят «расширение работает не так / документ упал / поручительство не списалось», я хочу за минуты дойти от симптома (документ № N) до причины (строка в модуле М) и подтвердить причину данными (запрос показывает что Y), чтобы выдать разработчику фикс-задачу с confidence ≥ 85% — а не «попробуй посмотреть».*

**Реальная боль:**
- Эп.4: 5 BSL-модулей ~3000 строк через MCP, 12 CRITICAL вручную, confidence scoring пальцем в небо
- Эп.7: 32 документа ОПП за 30.04 без шапки — `ЗаполнитьЗначенияСвойств` молча игнорирует type-mismatch. Это не видно ни в логе, ни в журнале регистрации
- Эп.17: 13 багов в собственном инструменте — пришлось делать рефакторинг чтобы их найти

**Что должен делать инструмент:**
- «Я знаю что документ № X ведёт себя неправильно. Покажи путь: что меняли, какие подписки сработали, кто звал ПриЗаписи»
- Stack-trace из журнала регистрации → ссылка на строку BSL → ссылка на регистр данных
- Validation hooks: после массового создания документов — авто-сверка «какие реквизиты остались Неопределено»
- Bug-list с confidence score (как в QA-аудите 2026-05-05) + drill-down к каждому
- Repro-data: для каждого бага — минимальный SQL-запрос показывающий N документов с этим симптомом

### US-3. Drill-down цифры из отчёта руководству

**Сырые эпизоды:** 5, 6, 9, 12.

**История:** *Когда руководитель / заказчик показывает мне цифру («у нас 4197 документов в плохих статусах», «оборот -12% по сравнению с прошлым месяцем», «79 работ в конструкторе аудита»), я хочу за минуты декомпозировать её до строк-источников (какие именно документы, какие регистры, какие проводки) и обратно собрать в график — чтобы дать ответ той же встречи, а не «через неделю когда разработчик сделает отчёт».*

**Реальная боль:**
- Эп.9: 62% НеРассчитан, 31% НеИдентифицирован — собирал SQL-запросами через MCP по одному
- Эп.5: 296 дублей НомерГТД — узнал только когда прогнал реальный CSV, не было предупреждения
- Эп.6: приёмка по 8 счётчикам — считал глазами по форме
- Эп.12: конструктор 79 работ — Excel автопересчёт, но согласовывать нельзя построчно

**Что должен делать инструмент:**
- Notebook-style: запрос → таблица → группировка → drill-down кликом — в одной странице
- AI-генерация запроса по NL: «сколько ОПП за апрель в статусе НеРассчитан с заполненным Декларантом»
- «Объясни число»: AI читает запрос + результат и пишет TL;DR для руководителя
- Save-as-snapshot: на любую цифру можно сослаться через permalink + комментарии
- Compare: «эта цифра сегодня / неделю назад / месяц назад»

### US-4. Маппинг внешних данных в 1С

**Сырые эпизоды:** 2, 5, 7.

**История:** *Когда мне приносят CSV / выгрузку API / Excel и просят «загрузи в 1С как документы», я хочу за один проход увидеть mapping (поле X → реквизит Y), увидеть конфликты (дубликаты, несоответствие типов, неизвестные значения справочников) и подтвердить корректность данными — чтобы не узнавать о 296 дублях после прогона и не получать 32 документа без шапки.*

**Реальная боль:**
- Эп.7: type-mismatch String(10) ↔ СправочникСсылка → `ЗаполнитьЗначенияСвойств` молча игнорирует → 32 битых документа
- Эп.2: парсинг CSV в несколько итераций (двойной CR, многострочные поля, точка с запятой внутри кавычек)
- Эп.5: 296 дублей НомерГТД в БД — отдельный лог нужен

**Что должен делать инструмент:**
- Wizard: «вот CSV → вот колонки → сопоставь с реквизитами / создай новые». Превью первых 10 строк перед прогоном
- Dry-run: показать «будет создано 437 документов, из них 78 новых ТД, 359 апдейтов, 0 пропусков»
- Validation report: какие записи не пройдут (тип несовместим, обязательный реквизит пуст, контрагент не найден)
- Сравнение источников: «CSV vs БД vs API Сигмы» — показать строки только в одном из трёх
- Reusable mapping profiles: маппинг для регулярных загрузок сохраняется и переиспользуется

### US-5. Согласование с заказчиком / requirements collaboration

**Сырые эпизоды:** 3, 8, 11, 12.

**История:** *Когда я веду требования с заказчиком (встреча, чат, документ обследования), я хочу одну точку правды где живут вопросы, ответы, открытые гипотезы, дисcuss-этого-на-след-встрече — а не Telegram-чат + Excel-вопросник + .docx-отчёт + 68 разрозненных документов — чтобы перед встречей открыть «что не закрыто» за 30 секунд.*

**Реальная боль:**
- Эп.3: 5 подготовленных вопросов на встрече не успели — нет «синхронной повестки» которую видно обоим
- Эп.8: 68 документов клиента, требования разбросаны → собирал в один requirements.json вручную
- Эп.11: 168-страничный отчёт об обследовании Век — структурного diff'а с типовой методологией нет
- Эп.12: 79 работ × 10 блоков в Excel — нет комментариев построчно

**Что должен делать инструмент:**
- Дерево требований (FR/NFR/IR), MoSCoW-priority, статусы (черновик / согласован / отвергнут)
- Каждое требование → ссылки на источники (документ, встреча, цитата заказчика)
- Open-questions panel: всё что «надо спросить на след. встрече»
- Comments thread на каждом требовании — для аналитика и заказчика
- Templates: «методология обследования (ГОСТ / 1С:ПМ / ТКВ)» — diff с тем что собрали

### US-6. Knowledge work: knowledge base + research

**Сырые эпизоды:** 13, 14, 16 (плюс мета — эп. 15).

**История:** *Когда я работаю над конфигурацией / темой, я постепенно нахожу ловушки, паттерны, инструменты — я хочу складывать находки в knowledge base со связями (конфигурация → ловушка → пример → проект) и быстро находить «у меня уже было нечто похожее в Веке, вот как решил» — а не дублировать research каждый раз.*

**Реальная боль:**
- Эп.13: 18 ловушек P-001..P-018 по УСО — лежат в одном .md, но не связаны с реальными проектами
- Эп.14: 12 поисков по рынку инструментов миграции — для каждой новой темы повторяется
- Эп.16: research 8 MCP-серверов + 3 skill-сетов — где это будет лежать через полгода?

**Что должен делать инструмент:**
- Knowledge graph: конфигурация ↔ объект ↔ ловушка ↔ проект где встретилось
- Tag system: «УТ 11.5», «ERP УСО 2.5», «КА 2.5» + «политики», «миграция», «обмен»
- Search-as-you-type: «миграция остатков УТ→БП» → находит «есть бесплатная ИТС-обработка»
- Capture-from-anywhere: из чата с AI / из drill-down цифры / из bug-triage → «сохрани как заметку»
- Periodic resurfacing: «3 месяца назад ты находил X, сейчас обсуждаешь Y — может релевантно?»

## Что НЕ попало в user-stories (явно out of scope)

- **Деплой расширения / EDT workflow** — это работа разработчика, не аналитика
- **Написание BSL-кода руками** — юзер этого не делает (читает / проектирует / ставит задачи)
- **CI/CD, тесты, security** — есть отдельные инструменты, аналитику не нужны в одном workspace
- **Telegram-бот / e-mail integration** — добавляет communication, но не решает 6 главных болей
