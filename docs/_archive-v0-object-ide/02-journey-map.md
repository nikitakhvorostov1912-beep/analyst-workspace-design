# Phase 2 — Journey Map

**Дата:** 2026-05-13
**Метод:** реконструкция типичной недели P1 (Никита) на основе 17 эпизодов из Phase 0. Маппинг шагов на 10 MCP tools + слои 2/3 архитектуры.

---

## Контекст: типичная рабочая неделя P1

**4-6 параллельных проектов одновременно.** Не «делаю один проект до сдачи» — переключения по 2-4 раза в день. День в среднем 6-7 рабочих часов, из них:
- ~50% — глубокая работа на одном проекте (1.5-2 ч блоки)
- ~25% — встречи (клиенты, разработчики)
- ~15% — переключения / контекст-разогрев между проектами
- ~10% — knowledge work (заметки, поиск, обучение)

**Главный враг:** контекст-свитчинг. Тратится не время на переключение — тратится **точка входа в проект** (где остановился, что было обещано, что не закрыто).

```
       ▲
   ████│
   ████│           ▲
   ████│          ╱ ╲
   ████│         ╱   ╲      ▲
   ████│  ▲     ╱     ╲    ╱ ╲

   ████│╱   ╲ ╱         ╲╱     ╲___
   ████│     V
       └────┬──┬──┬──┬──┬──┬──┬─►
            09 11 12 13 14 15 17  время
            ↑     ↑       ↑       ↑
            старт обед    встреча drill-down
                  ↓               под усталость
              ВРАГ — контекст-свитчинг
```

**Где «пьёт энергию» (-)**:
- 09:00-09:30 → утренний triage по чатам (рутина, ничего нового)
- 11:30-12:30 → переключение РТ → Век без savedState
- 14:30-15:30 → встреча требует «собрать всё что обсудим» (контекст)

**Где даёт энергию (+)**:
- Bug-triage когда найдена причина за 30 мин — высокий «ага»
- Discovery новой базы — это интересно само по себе (хобби+работа)

**Что инструмент должен УСИЛИТЬ:**
- Скорость старта дня — «открыл, увидел что важно, поехал»
- Сохранение состояния проекта — «закрыл вкладку Век, открыл — ровно где остановился»
- Связи между объектами — клик от «32 ОПП без шапки» к «реквизит ОсновноеТС» к «модуль APIСигмы»

**Что инструмент должен МИНИМИЗИРОВАТЬ:**
- Терминал / скрипты / копирование UUID
- Переключение между Excel + Telegram + EDT + MCP скрипт
- Ручное форматирование (Markdown vs Excel vs Word)

## Пиковая нагрузка: discovery незнакомой базы (день 1 проекта)

Это **самая дорогая активность для аналитика**. Сейчас — 6-8 часов растянуто на 1-2 дня. С инструментом — 2-3 часа.

Sequence (для P1 и для P2 одинаково):

```
1. Подключить MCP к базе клиента (?channel=client-N)
   └─ Connect wizard, проверка живого MCP

2. Карта базы (get_metadata summary)
   └─ «Справочников 265, Документов 27, Регистров 150»

3. Heatmap активности (последний месяц)
   └─ execute_query по системным регистрам / журналу
   └─ «активно: Реализация (1450), Поступление (1100), пусто: Доверенность (0)»

4. Список расширений (get_metadata extension_name="")
   └─ «5 расширений: Транзит, Сигма, ИзменениеЦен, ...»

5. Для каждого расширения — что переопределено
   └─ get_metadata extension_name=X meta_type=*
   └─ matrix «расширение × объект → перехвачено / новый»

6. Ловушки (если конфигурация известна — например УСО)
   └─ для каждой P-001..P-018 → execute_query → результат
   └─ «найдено: P-003 (синхронные диалоги в EPF), P-010 ...»

7. Сформированный отчёт «День 1 — обзор базы клиента N»
   └─ markdown + ссылки на объекты + список вопросов клиенту
```

**Каждый шаг — отдельная MCP-операция. Workspace их оркеструет.**

## Открытые вопросы для Phase 3 (IA)

1. **«Проект» как организационная единица?** — каждый клиент / каждая база = отдельный проект с savedState, заметками, requirements
2. **Inbox-view нужен?** — или достаточно «список проектов с индикатором что есть новое»
3. **Live notes во время встречи** — это та же поверхность что и requirements или отдельный режим?
4. **Heatmap активности** — это auto-job (фоновое) или on-demand?
5. **Auto-check ловушек** для известной конфигурации — это feature MVP или потом?
