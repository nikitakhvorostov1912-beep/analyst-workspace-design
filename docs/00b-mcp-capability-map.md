## 10 инструментов MCP Toolkit — паспорт

### 1. `get_metadata` — карта структуры базы

5 режимов в одном tool'е:

| Режим | Вызов | Что даёт |
|-------|-------|----------|
| Summary | без параметров | счётчики корневых типов: «Справочников 265, Документов 27, РегистровСведений 150» |
| List | `meta_type` + `name_mask` | список объектов: «все документы со словом 'реализ'» |
| Detail | `filter="Документ.X"` + `sections` | полная карточка: реквизиты, ТЧ, формы, команды, макеты, движения, характеристики |
| Collection element | `filter="...Реквизит.ИНН"` | один реквизит/измерение/ТЧ-реквизит |
| Attribute search | `attribute_mask="контраг"` | поиск реквизита по имени/синониму ВО ВСЕХ объектах |
| Extension scope | `extension_name=""` или `"name"` | список расширений или объекты внутри расширения |

**Ключевая фишка:** `attribute_mask` — это «найди все объекты у которых есть реквизит со словом X». Прямой ответ на «куда воткнут Декларант».

### 2. `execute_query` — запросы 1С

- Текст запроса 1С + `params` (key-value) + `limit` (1-1000)
- `include_schema=true` → типы колонок в ответе (без загрузки данных: `ВЫБРАТЬ ПЕРВЫЕ 0 *`)
- Ссылки в результатах автоматически = `object_description` (можно передавать в другие tool'ы)

### 3. `execute_code` — произвольный BSL

- `execution_context: "server"` (по умолч., `&НаСервереБезКонтекста`) или `"client"` (`&НаКлиенте`, UI)
- Результат через `Результат = ...;` (без `Возврат`)
- **Blacklist опасных слов:** `Удалить`, `Записать`, `УстановитьПривилегированныйРежим`, `COMОбъект`, `УдалитьФайлы`, `КопироватьФайл` и др. Настраивается через `ALLOW_DANGEROUS_WITH_APPROVAL` — тогда требует подтверждения в UI 1С

### 4. `get_object_by_link` — полный объект по навигационной ссылке

`e1cib/data/Справочник.X?ref=HEX32` → весь объект с реквизитами и ТЧ

### 5. `get_link_of_object` — обратное: object_description → ссылка

Для построения ссылок «открой документ N» из результата запроса

### 6. `find_references_to_object` — где используется объект

- `target_object_description` (UUID + тип) + `search_scope` (`documents` / `catalogs` / `information_registers` / `accumulation_registers` / `accounting_registers` / `calculation_registers`)
- `meta_filter` (names или name_mask), `limit_per_meta`, `timeout_budget_sec`
- Возвращает hits с указанием **в каком объекте, в каком пути** (атрибут/ТЧ/измерение/ресурс), плюс представление

### 7. `get_access_rights` — права на объект

- По объекту метаданных → роли + права + опционально effective rights пользователя
- **Ограничение:** RLS и контекстные ограничения **не учитываются** — это «сумма ролей»

### 8. `get_event_log` — журнал регистрации

Фильтры (можно комбинировать):
- `start_date` / `end_date` (ISO 8601)
- `levels` (Error / Warning / Information / Note)
- `events` (`_$Data$_.New`, `_$Data$_.Update` и т.д.)
- `object_description` или `link` или `metadata_type` — для одного объекта или типа
- `user`, `session`, `application` (ThinClient/WebClient/BackgroundJob/HTTPService и т.д.)
- `comment_contains` (substring)
- `transaction_status` (Committed/RolledBack/NotApplicable/Unfinished)
- Курсорная пагинация через `last_date` + `next_same_second_offset`

### 9. `get_bsl_syntax_help` — справочник встроенного языка

Полнотекстовый поиск + breadcrumb-paths (типа `Массив/Методы/Найти`). Возвращает Markdown с синтаксисом и параметрами. Пагинируется по контенту для больших страниц (Запрос/ТаблицаЗначений).

### 10. `submit_for_deanonymization` — финальный ответ для разанонимизации

Только если анонимизация включена. Возвращает `{"received": true}`.

## Бонусные возможности (не отдельные tool'ы, но важны)

- **Анонимизация:** реальные значения → токены `[ORG-00001]`, `[INN-00001]`, `[PER-00001]`. Можно слать в LLM без PII. Управляется через форму EPF (дерево полей / словари / regex) или env vars в proxy-режиме
- **Изоляция каналов:** `?channel=dev` vs `?channel=prod` — параллельно работать с несколькими базами / окружениями
- **TOON формат** (Token-Oriented Object Notation) — компактное представление таблиц для LLM (3-5× меньше токенов чем JSON), выбирается в EPF
- **REST API** параллельно с MCP — те же endpoints через `/api/*` для интеграции с FastAPI/Next.js без MCP-обёртки

## Покрытие 6 user-stories возможностями MCP

| US | Покрытие | Главные tool'ы | Что НЕ покрыто (вне MCP) |
|----|----------|----------------|--------------------------|
| **US-1 Discovery незнакомой базы** | **95%** | `get_metadata` (5 режимов!) + `find_references_to_object` + `get_event_log` (что движется) | Bookmarks/notes/диаграммы — UI-слой |
| **US-2 Bug-triage код→данные** | **90%** | `get_event_log` + `execute_query` + `get_object_by_link` + `find_references_to_object` + `get_bsl_syntax_help` | Чтение BSL-исходников расширения — это файлы CFE, не MCP. Confidence scoring — LLM-слой |
| **US-3 Drill-down цифры** | **85%** | `execute_query` (с `include_schema`) + `get_metadata` для подсказки реквизитов | NL→SQL генерация — LLM-слой. Графики/таблицы — UI |
| **US-4 Маппинг внешних данных** | **50%** | `get_metadata` (структура целевого) + `execute_code` (создание объектов) + `execute_query` (дедуп-проверка) | Парсинг CSV/Excel — отдельный pipeline. Wizard mapping — UI |
| **US-5 Согласование с заказчиком** | **5%** | `get_metadata` для «вот так это в типовой» | Это коммуникация, не данные. Полностью UI/DB слой |
| **US-6 Knowledge work** | **20%** | `get_bsl_syntax_help` (BSL only) | Knowledge graph по проектам/конфигурациям — отдельная подсистема |

**Вывод:** MCP — это «движок к живой 1С». Покрывает **US-1, US-2, US-3 почти полностью** (то самое «80% боли»). US-4/5/6 нуждаются в собственном слое поверх MCP.

## Архитектурная развилка: три слоя workspace

```
┌─────────────────────────────────────────────────────────────┐
│  СЛОЙ 3 — Knowledge & Collaboration                          │
│  Bookmarks, заметки по объектам, requirements-tree,          │
│  open-questions, knowledge graph по проектам                 │
│  Storage: SQLite/Postgres локально + опционально шарить      │
│  → закрывает US-5, US-6                                       │
├─────────────────────────────────────────────────────────────┤
│  СЛОЙ 2 — Pipeline & Workflow                                │
│  CSV/Excel parser, mapping wizard, dry-run validator,        │
│  bug-list aggregator, drill-down notebooks                   │
│  → закрывает US-4 + дополняет US-2/US-3                       │
├─────────────────────────────────────────────────────────────┤
│  СЛОЙ 1 — MCP Adapter (тонкий)                               │
│  Обёртка над 10 MCP tools, кэширование metadata,             │
│  построение запросов из NL, формирование TOON-ответов        │
│  → закрывает US-1, US-2, US-3                                 │
├─────────────────────────────────────────────────────────────┤
│  СЛОЙ 0 — MCP Toolkit (готовый)                              │
│  EPF в 1С, порт 6010/6003, 10 tools                          │
└─────────────────────────────────────────────────────────────┘
```

**Что это значит для дизайна workspace:**

- **Главный экран = карта базы** на `get_metadata`-summary. Drill-down кликом → list → detail. Это US-1 в чистом виде, ноль кода
- **Командная строка / NL-input** → execute_query/execute_code, ответ + представление. Это US-3
- **«Найти X»** → attribute_mask + find_references. Это US-1/US-2 повседневно
- **Журнал** как первоклассный объект (фильтры всегда видны) — US-2 «что упало, кто менял»
- **Workspace объекта** (открыл документ ОПП): карточка + права + где используется + история (event log по нему) + связанные регистры. 4 tool'а в один экран
- **Wizard загрузки** — отдельный режим с пайплайном (Layer 2)
- **Заметки / requirements** — отдельный режим (Layer 3)

## Концептуальные следствия для Phase 4 (3 альтернативы)

Подсказки концептам:

- **A. Notebook-first** — каждая ячейка = MCP call. Запрос → таблица → drill-down → AI-объяснение. Идеально для US-3
- **B. Object-centric IDE** — слева дерево метаданных (get_metadata списком), посередине карточка объекта (detail+references+events), справа AI-чат с контекстом текущего объекта. Идеально для US-1/US-2
- **C. Chat-first** — основной интерфейс — диалог с AI, который под капотом дёргает MCP. Каждый ответ AI содержит «прикладные карточки» (объект / список / число / лог-запись)

Эти три концепта рождаются НАПРЯМУЮ из mapping'а tool'ов к user-stories. Это не маркетинг — это разные принципы доступа к 10 MCP-операциям.

## Открытые вопросы для Phase 1

1. **Multi-tenant с самого начала?** — каналы (`?channel=`) позволяют работать с базами 5-10 клиентов параллельно. Это сразу или потом?
2. **Анонимизация — ON или OFF по умолчанию?** для юридически чистого отправления контекста в LLM (см. правило в `1c-ai-collaboration.md`)
3. **Где BSL-исходники?** MCP даёт **код метаданных** через `get_metadata`, но не модули. Чтение модулей расширения — отдельный канал (EDT MCP / файлы / git). Это для US-2 критично — нужно решить откуда брать
4. **Кэш metadata** локально? — `get_metadata` для большой конфигурации может быть медленным. Кэш с TTL = быстрый отклик в UI
5. **«Подсказчик запросов»** — между NL-input и `execute_query` нужен слой который читает `get_metadata` + `get_bsl_syntax_help` и генерирует корректный запрос. Это самый ценный AI-блок системы
